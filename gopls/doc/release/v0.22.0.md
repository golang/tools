## Analysis features

### `atomictypes` analyzer
<!-- golang/go#77352 -->

The new atomictypes analyzer suggests replacing the primitive `sync/atomic` functions with
the strongly typed atomic wrapper types introduced in Go1.19 (e.g.
atomic.Int32). It refactors declarations like `var x int32` to
`var x atomic.Int32` and updates all corresponding call sites from
`atomic.AddInt32(&x, 1)` to method calls like `x.Add(1)`.

### `fieldassignment` analyzer
<!-- golang/go#76237 -->
The fieldassignment analyzer, which reports diagnostics about memory layout and
was disabled in v0.17.0, has been restored. It is disabled by default.

### `scannererr` analyzer
<!-- golang/go#17747 -->
The scannererr analyzer reports failure to call the Err method when
calling the Scan method of bufio.Scanner in a loop.

### `writestring` analyzer
<!-- golang/go#77369 -->

The new writestring analyzer detects inefficient string concatenation in uses of WriteString,
of the form WriteString(a+b), and offers to replace them with WriteString(a); WriteString(b).

### `waitgroup` modernizer renamed to `waitgroupgo`
<!-- golang/go#77560 -->

Previously there were two analyzers named `waitgroup`: an analyzer that checks
for misuses of `sync.WaitGroup`, and a modernizer that simplifies goroutine
management with `sync.WaitGroup`. We have renamed the modernizer to `waitgroupgo`.

## Code transformation features

### Interactive code actions
<!-- golang/go#76331 -->

Support has been added for interactive refactorings that prompt the user for input before proceeding. This enables more sophisticated code actions that require user choices.

This release includes three interactive code actions:
- **Add tags to struct fields**: Prompts the user for the tags to add and the casing transformation rule (e.g., `camelCase`, `snake_case`).
- **Remove tags from struct fields**: Prompts the user for the tags to remove.
- **Implement interface**: A new code action that adds method stubs to a type to implement a specified interface.

Dialog support is a non-standard feature of LSP. It is currently supported only by `gopls` and requires a language client that supports the proposed protocol (such as VS Code). We plan to work with the broader LSP community toward a standard specification.
